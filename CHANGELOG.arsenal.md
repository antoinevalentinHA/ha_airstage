# Arsenal fork changelog

Changes specific to the `antoinevalentinHA/ha_airstage` fork, branch `arsenal-stable`.
Upstream `CHANGELOG.md` remains untouched and is generated by semantic-release.

## [arsenal-2026.05.20] — 2026-05-20

### Fixed

- **setup semantics:** Replace `ConfigEntryError` with `ConfigEntryNotReady` on
  both cloud and local branches of `async_setup_entry`. Transient device
  unavailability at boot is no longer logged as a permanent ERROR; Home
  Assistant now retries automatically with backoff per its documented
  convention for transient setup failures.
- **error message honesty:** Drop the misleading "Timeout while connecting"
  wording on the local branch. `pyairstage.ApiError` covers more than
  timeouts (`No valid response`, server-disconnect, payload errors), so the
  prior message was often wrong. New message names only the relevant
  identifiers (`device_id`, `ip_address`) and includes the underlying error.
- **log hygiene:** Stop dumping the full `entry.data` and `entry.options`
  dicts in the error message.

### Changed

- **boot tolerance:** Bump `AIRSTAGE_LOCAL_RETRY` from 3 to 5 to better absorb
  the case where the AC unit is still waking up when Home Assistant
  initializes the integration. Underlying `pyairstage` per-request timeout
  (10 s) is unchanged.

### Notes

- `async_config_entry_first_refresh()` is intentionally preserved on both
  branches. `entity.py` accesses `coordinator.data[ac_key]` at entity
  construction time (lines 55–56, 64), so a populated first refresh remains
  a hard prerequisite.
- No behavior change when the device is reachable.
